# Техническое задание
## Telegram-бот для планирования и заказа корпоративного водителя

**Версия:** 1.0
**Дата:** 22.07.2026
**Статус:** Черновик для согласования

---

## 1. Общее описание

### 1.1. Назначение системы
Telegram-бот (с встроенным Mini App) для внутреннего использования компанией: сотрудники бронируют поездки с корпоративным водителем, водители видят своё расписание и заказы, администраторы верифицируют сотрудников и управляют системой. Все три роли работают внутри одного бота.

### 1.2. Целевая аудитория
Небольшой закрытый круг пользователей (ориентировочно до 100–200 сотрудников, 1–10 водителей, 1–3 администратора). Доступ только по верификации.

### 1.3. Ключевая ценность
- Сотрудник за 3–4 нажатия оформляет заказ поездки с расчётом примерного времени в пути.
- Водитель получает единое расписание без звонков и переписок.
- Админ контролирует, кто имеет доступ к сервису.

## 2. Стек технологий

### 2.1. Backend
| Компонент | Технология | Обоснование |
|---|---|---|
| Язык / фреймворк | **Python 3.12 + FastAPI** (альтернатива: Node.js + NestJS) | Быстрая разработка, асинхронность, автодокументация API (OpenAPI) |
| Telegram Bot | **aiogram 3.x** (или grammY для Node) | Современный асинхронный фреймворк, FSM «из коробки» |
| БД | **PostgreSQL 16** | Надёжность, транзакции, работа с временными интервалами (tstzrange, exclusion constraints против пересечения броней) |
| ORM / миграции | SQLAlchemy 2.0 + Alembic | Стандарт де-факто |
| Кэш / очереди | **Redis** | FSM-состояния бота, кэш геокодинга, rate limiting |
| Фоновые задачи | **Celery / ARQ** | Напоминания, авто-отмена неподтверждённых заказов, отчёты |
| Расчёт маршрута | **OpenRouteService / OSRM** (self-hosted) или **Yandex Maps API / 2GIS API** | Геокодинг адресов + матрица времени в пути. Для РФ рекомендуется Yandex/2GIS (точнее адреса и пробки) |

### 2.2. Frontend (Telegram Mini App)
| Компонент | Технология |
|---|---|
| Фреймворк | **React 18 + TypeScript + Vite** |
| UI-интеграция с Telegram | **@telegram-apps/sdk** (Telegram Mini Apps SDK) — тема, haptic feedback, MainButton, BackButton |
| Стилизация | **Tailwind CSS** + CSS-переменные для темы |
| Анимации | **Framer Motion** (переходы экранов, микроанимации), **Lottie** (иллюстрации состояний: успех, пустые списки, загрузка) |
| Карта | Yandex Maps JS API / MapLibre GL (выбор точки на карте, отображение маршрута) |
| Состояние | Zustand / TanStack Query (серверный кэш) |

### 2.3. Инфраструктура и DevOps
- **Docker + docker-compose** (bot, api, worker, postgres, redis, nginx).
- **Nginx** — reverse proxy, TLS (Let's Encrypt), раздача статики Mini App.
- Webhook-режим бота (не polling) для продакшена.
- **Sentry** — мониторинг ошибок; структурированные логи (JSON).
- CI/CD: GitHub Actions (линт, тесты, сборка образов, деплой).
- Бэкапы PostgreSQL: ежедневный pg_dump + хранение 30 дней.

## 3. Роли и авторизация

### 3.1. Модель авторизации
1. **Идентификация** — автоматически по `telegram_id` (получается из `initData` Mini App, подпись обязательно валидируется на бэкенде по HMAC с bot token; срок жизни initData ≤ 1 час).
2. **Привязка аккаунта** — при первом входе пользователь указывает корпоративную почту **или** телефон:
   - Почта: отправка 6-значного кода на email (SMTP), ввод кода в боте/Mini App. TTL кода — 10 минут, не более 5 попыток.
   - Телефон: кнопка Telegram «Поделиться контактом» (`request_contact`) — номер приходит верифицированным от Telegram; дополнительно возможна проверка по белому списку номеров сотрудников.
3. **Верификация** — после привязки аккаунт попадает в статус `pending`. Админ получает уведомление и подтверждает/отклоняет заявку. Только статус `verified` открывает доступ к заказам.
4. Роли назначаются админом: `user` (по умолчанию после верификации), `driver`, `admin`. Один пользователь может иметь только одну роль (админ при необходимости может также заказывать поездки — флаг `can_order`).

### 3.2. Роли

**Пользователь (сотрудник):**
- Создание, просмотр, изменение (до подтверждения), отмена своих заказов.
- Просмотр свободных слотов водителей.
- История поездок, повтор прошлого маршрута «в один клик».
- Избранные адреса («Офис», «Дом», «Аэропорт»).

**Водитель:**
- Лента своих заказов: новые → принятые → в пути → завершённые.
- Расписание на день/неделю (таймлайн занятости).
- Действия по заказу: принять, «выехал», «на месте», «поездка началась», «завершена»; отклонить с указанием причины.
- Установка своей доступности (рабочие часы, перерывы, выходные, отпуск).
- Кнопка «Открыть маршрут» — deeplink в Яндекс.Навигатор / Google Maps.

**Администратор:**
- Заявки на верификацию: список pending, карточка (ФИО, email/телефон, tg username), кнопки Одобрить / Отклонить (с причиной).
- Управление пользователями: смена роли, блокировка, разблокировка, удаление.
- Управление водителями: добавление, привязка авто (марка, номер, цвет), графики.
- Все заказы в системе: фильтры по дате/статусу/водителю, ручное назначение и переназначение водителя, принудительная отмена.
- Статистика: количество поездок за период, загрузка водителей, топ маршрутов; выгрузка в CSV/XLSX.
- Настройки: рабочие часы сервиса, горизонт бронирования (например, до 14 дней), минимальное время до поездки (например, 30 минут), буфер между поездками.

## 4. Функциональные требования

### 4.1. Сценарий заказа поездки (основной флоу)
1. Пользователь открывает Mini App → главный экран с кнопкой «Заказать поездку» и ближайшими заказами.
2. Шаг «Откуда»: избранные адреса / последние адреса / поиск с подсказками (геокодер) / выбор точки на карте.
3. Шаг «Куда»: аналогично.
4. Система запрашивает у маршрутного API **примерное время поездки и расстояние**, отображает: «≈ 35 мин · 18 км» и маршрут на карте.
5. Шаг «Когда»: выбор даты (календарь на горизонт бронирования) и времени. Отображаются **только свободные слоты** с учётом: занятости водителей, длительности поездки + буфер (настраиваемый, по умолчанию 15 мин), рабочих часов.
6. (Опционально) выбор конкретного водителя или «любой свободный»; комментарий к заказу; число пассажиров.
7. Экран подтверждения → создание заказа со статусом `pending_driver`.
8. Уведомление водителю (или всем свободным при «любой свободный» — заказ забирает первый принявший). После принятия — статус `confirmed`, пользователю приходит уведомление с данными водителя и авто.

### 4.2. Жизненный цикл заказа (статусная модель)
```
draft → pending_driver → confirmed → driver_en_route → driver_arrived
      → in_progress → completed
Ветки: → cancelled_by_user / cancelled_by_driver / cancelled_by_admin / expired
```
- `expired` — авто-отмена, если ни один водитель не принял заказ за N минут (настраивается) — уведомление админу и пользователю.
- Изменение заказа пользователем возможно до статуса `driver_en_route`; после — только отмена (с уведомлением водителя).
- Отмена позже чем за X минут до поездки помечается как «поздняя» (видна в статистике).
- Защита от пересечений: на уровне БД exclusion constraint по `driver_id + tstzrange(start, end + buffer)`.

### 4.3. Расчёт времени поездки
- Геокодинг адреса → координаты (кэш в Redis на 30 дней по нормализованной строке).
- Запрос времени в пути; при заказе на будущее — по возможности с учётом типичной загруженности на это время (Yandex Matrix API поддерживает).
- К результату применяется настраиваемый коэффициент запаса (по умолчанию ×1.2).
- Fallback: при недоступности API — оценка по прямому расстоянию (haversine × 1.4 / средняя скорость 30 км/ч) с пометкой «примерно».

### 4.4. Уведомления (через сообщения бота)
| Событие | Кому |
|---|---|
| Новая заявка на верификацию | Админ |
| Аккаунт верифицирован / отклонён | Пользователь |
| Новый заказ | Водитель(и) |
| Заказ принят / отклонён / отменён | Пользователь ↔ Водитель |
| Напоминание за 60 и за 15 мин до поездки | Пользователь и Водитель |
| Водитель выехал / на месте | Пользователь |
| Заказ не принят (expired) | Пользователь + Админ |

Все уведомления с inline-кнопками быстрого действия (например, «Принять / Отклонить» прямо в сообщении).

### 4.5. Команды бота (вне Mini App)
- `/start` — приветствие, онбординг, кнопка открытия Mini App.
- `/help` — справка по ролям.
- `/myorders` — краткий список ближайших заказов (для пользователя) / на сегодня (для водителя).
- Ролевое меню-кнопка (Menu Button) открывает соответствующий раздел Mini App.

## 5. Требования к интерфейсу (UI/UX)

### 5.1. Общие принципы
- **Mini App — основной интерфейс**, чат бота — для уведомлений и быстрых действий.
- Мобайл-фёрст, один экран — одна задача, крупные тач-зоны (min 44 px).
- Поддержка светлой и тёмной темы Telegram (цвета через CSS-переменные, синхронизация с `themeParams`).
- Скелетоны при загрузке, пустые состояния с Lottie-иллюстрациями, haptic feedback на ключевых действиях.

### 5.2. Дизайн-система
**Палитра (акценты: фиолетовый, синий, белый):**
| Токен | Цвет | Использование |
|---|---|---|
| `primary` | #7C3AED (фиолетовый) | Основные кнопки, активные элементы, акценты |
| `secondary` | #3B82F6 (синий) | Ссылки, второстепенные действия, статусы «в пути» |
| `gradient` | linear-gradient(135°, #7C3AED → #3B82F6) | Хедеры карточек, главная кнопка, прогресс |
| `bg` | #FFFFFF / #0F0F1A (dark) | Фон |
| `surface` | #F5F3FF / #1C1B2E (dark) | Карточки |
| `success / warning / danger` | #10B981 / #F59E0B / #EF4444 | Статусы |

- Типографика: Inter (или системный SF/Roboto), шкала 12/14/16/20/24.
- Скругления 16–20 px, мягкие тени, glassmorphism допустим для хедера.
- Статусы заказов — цветные бейджи с иконками.

### 5.3. Анимации
- Переходы между экранами — слайд/fade (Framer Motion, 200–250 мс).
- Появление карточек списка — staggered fade-in.
- Прогресс заказа — анимированный степпер (заказ → принят → в пути → завершён).
- Успешное создание заказа — полноэкранная Lottie-анимация (галочка/машинка).
- Все анимации ≤ 300 мс, уважение `prefers-reduced-motion`.

### 5.4. Ключевые экраны
1. **Онбординг/привязка** — 2–3 слайда, ввод email/телефона, ввод кода, экран «Ожидает верификации».
2. **Главный (пользователь)** — приветствие, ближайшая поездка карточкой с картой-превью, кнопка «Заказать», история.
3. **Мастер заказа** — 4 шага со степпером (адреса → карта/время в пути → дата/время слоты → подтверждение).
4. **Карточка заказа** — маршрут на карте, статус-степпер, данные водителя/авто, кнопки действий.
5. **Водитель: сегодня** — вертикальный таймлайн поездок, свайп-действия, переключатель доступности.
6. **Админ-панель** — табы: Заявки (с бейджем количества) / Пользователи / Заказы / Статистика (графики) / Настройки.

## 7. Нефункциональные требования

### 7.1. Безопасность
- Валидация подписи `initData` на каждом запросе к API; далее — короткоживущий JWT (15 мин) + refresh.
- Все данные и роли проверяются **на бэкенде**; фронт лишь скрывает недоступные разделы.
- HTTPS everywhere; секреты — в переменных окружения / vault, не в репозитории.
- Rate limiting: запросы к API (30/мин на пользователя), отправка кодов (3 в час).
- Хранение ПДн (ФИО, телефон, email) — минимально необходимое, шифрование бэкапов; соответствие 152-ФЗ (если РФ): согласие на обработку при онбординге, хранение на серверах в РФ.
- Аудит-лог всех действий админа и смен статусов заказов (`order_events`).

### 7.2. Производительность и надёжность
- Ответ API ≤ 300 мс (p95) без учёта внешних гео-API; расчёт маршрута ≤ 2 с.
- Доступность 99% (внутренний сервис); graceful degradation при падении гео-API (fallback-оценка).
- Идемпотентность создания заказа (клиентский idempotency key) — защита от даблкликов.
- Все времена в БД — UTC; отображение — в таймзоне компании (настройка).

### 7.3. Качество кода и тестирование
- Покрытие unit-тестами бизнес-логики ≥ 70% (слоты, пересечения, статусные переходы).
- Интеграционные тесты API; e2e основного флоу заказа.
- Линтеры/форматтеры (ruff/eslint + prettier), type-checking (mypy/tsc) в CI.

## 8. Этапы реализации

| Этап | Содержание | Оценка |
|---|---|---|
| 1. MVP | Бот + Mini App: привязка email/телефон, верификация админом, создание заказа, расчёт времени, принятие водителем, уведомления, базовые статусы | 4–5 недель |
| 2. Расписание | Графики водителей, слоты, буферы, авто-подбор «любой свободный», напоминания | 2 недели |
| 3. Админка+ | Статистика, выгрузки, настройки, аудит | 1–2 недели |
| 4. Полировка | Анимации, Lottie, тёмная тема, e2e-тесты, нагрузочное, деплой в прод | 1–2 недели |

**Итого: ~8–11 недель** (1 backend + 1 frontend разработчик).

### Критерии приёмки MVP
- Новый сотрудник проходит привязку и верификацию ≤ 2 минут (без ожидания админа).
- Полный цикл заказа (создание → принятие → завершение) работает с корректными уведомлениями всем сторонам.
- Невозможно создать два заказа на одного водителя с пересекающимся временем (включая буфер).
- Mini App корректно работает на iOS и Android в светлой и тёмной теме.

## 9. Риски и открытые вопросы
1. **Выбор гео-провайдера**: Yandex Maps API платный после лимита — уточнить ожидаемый объём запросов; альтернатива — self-hosted OSRM (бесплатно, но без пробок).
2. Нужна ли оплата/учёт стоимости поездок или сервис полностью внутренний?
3. Один водитель на заказ или возможны совместные поездки (карпулинг)?
4. Требуется ли трекинг геопозиции водителя в реальном времени (усложняет MVP)?
5. Политика при отпуске/болезни единственного водителя — уведомлять пользователей об отмене автоматически?
